One task inside the orchestrator

What to look at: the new Verify state actually EXECUTES the repo's tests before any reviewer reads the diff — a red verify never enters the gate — and a clear commit now pushes to the published branch.

Who talks to whom

What to look at: the driver still only sees one-line results; verify and push live inside the orchestrator, and the specialist security reviewer joins only at the integration step.

sequenceDiagram
    participant U as user
    participant D as dev-loop driver
    participant O as orchestrator
    participant R as review gate
    U->>D: /dev-loop feature
    D->>D: ensure worktree, publish branch
    loop one fresh agent per unchecked task
        D->>O: brief with seam map + locked decisions
        O->>O: implement, lean mode, unstaged
        O->>O: verify — RUN the tests, red never enters review
        O->>R: locked gate on the diff
        R-->>O: VERDICT PASS or FAIL, max 2 fix cycles
        O->>O: commit + tick plan + rewrite progress, then push
        O-->>D: ORCHESTRATOR RESULT, one line
    end
    D->>D: FULL test suite, red = FAIL
    D->>R: integration review, adds security axis
    R-->>D: PASS
    D->>U: show log, wait for explicit FF OK
    U->>D: OK
    D->>U: FF-merge, then ask about cleanup
